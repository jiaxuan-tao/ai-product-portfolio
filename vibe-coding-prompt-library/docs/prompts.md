# 提示词模板｜Vibe Coding 提示词资料库

本文档记录本地应用内置的 8 个默认提示词模板。

## PRD 初稿生成提示词

- 分类：PRD 提示词
- 适用场景：把一个初始产品想法整理成适合轻量项目推进的 PRD。

```text
请基于以下产品想法，整理一份轻量 PRD。

要求：

1. 不要写成商业包装或宣传文案。
2. 重点说明产品定位、目标用户、核心场景、MVP 功能、非目标范围和后续迭代方向。
3. 输出结构包括：
    * 背景
    * 用户场景
    * 产品目标
    * 最小功能范围
    * 核心用户流程
    * AI 能力
    * 暂不做什么
    * 后续迭代

产品想法：
[在这里粘贴产品想法]
```

## Codex 开发执行指令

- 分类：Codex 开发指令
- 适用场景：给 Codex 提供清晰的代码修改范围、目标和限制。

```text
请基于以下需求修改当前项目代码。

要求：

1. 先阅读项目结构，判断应该修改哪些文件。
2. 保持当前技术栈，不要引入不必要的新框架。
3. 优先做最小可用版本，不要扩大功能范围。
4. 修改完成后检查页面是否可以正常运行。
5. 如果涉及文案和 UI，请保持简洁、清晰、轻量。
6. 完成后只总结修改内容，不要提交，不要推送。

项目目标：
[填写项目目标]

需要修改的内容：
[填写具体修改内容]

不要修改：
[填写不希望改动的文件或功能]
```

## UI 优化提示词

- 分类：UI 优化
- 适用场景：在不重构功能逻辑的前提下优化页面视觉与基础适配。

```text
请优化当前页面 UI，但不要重构功能逻辑。

优化方向：

1. 提升页面层级和留白。
2. 优化字体、按钮、卡片、输入框和列表的视觉一致性。
3. 保持整体风格简洁，不要做成营销型落地页。
4. 不要引入复杂动效。
5. 不要改变核心交互流程。
6. 检查移动端和桌面端的基础适配。

当前问题：
[描述当前 UI 问题]

期望风格：
[描述期望风格]
```

## 问题排查与修复提示词

- 分类：问题修复
- 适用场景：描述项目问题、复现路径和期望结果，并要求做最小修复。

```text
请帮我修复当前项目中的问题。

请按以下步骤处理：

1. 先复现或定位问题来源。
2. 判断问题属于代码逻辑、样式、数据存储、状态管理还是运行环境。
3. 做最小必要修改，不要顺手重构无关代码。
4. 修复后说明修改了哪些文件。
5. 检查是否引入新的问题。
6. 完成后只总结修改内容，不要提交，不要推送。

问题描述：
[填写问题现象]

复现步骤：
[填写复现路径]

期望结果：
[填写正确表现]
```

## 项目 README 编写提示词

- 分类：README 编写
- 适用场景：为一个轻量 Vibe Coding 项目整理公开项目 README。

```text
请为这个 Vibe Coding 项目整理一份 GitHub README。

要求：

1. 不要写成宣传化表达。
2. 不要写成商业宣传。
3. 像公开项目记录，说明这个项目为什么做、解决什么小需求、如何实现、如何运行。
4. 如果项目基于开源项目二改，需要明确写出参考来源。
5. 输出结构包括：
    * 项目简介
    * 背景与目标
    * 核心功能
    * 使用场景
    * 技术结构
    * 本次改造范围
    * 运行方式
    * 后续计划
    * 参考来源

项目信息：
[粘贴项目说明]
```

## 用户流程整理提示词

- 分类：用户流程
- 适用场景：为轻量产品整理从进入页面到完成核心任务的用户流程。

```text
请为以下轻量产品整理用户流程。

要求：

1. 不要写成复杂业务流程。
2. 重点描述用户从进入页面到完成核心任务的路径。
3. 每一步说明用户动作、系统反馈和可能卡点。
4. 输出适合放入 GitHub 项目的 user-flow.md。

产品名称：
[填写产品名称]

核心任务：
[填写用户要完成的任务]

功能范围：
[填写当前版本功能范围]
```

## 产品结构整理提示词

- 分类：产品结构
- 适用场景：为轻量项目整理页面结构、模块结构和提示词相关能力。

```text
请为以下项目整理产品结构说明。

要求：

1. 说明页面结构、核心模块、数据结构和 AI / 提示词相关能力。
2. 不要写成技术架构说明书。
3. 重点体现产品模块如何支撑用户完成任务。
4. 输出适合放入 GitHub 项目的 product-structure.md。

项目名称：
[填写项目名称]

当前功能：
[填写功能列表]

技术实现：
[填写技术栈或实现方式]
```
